# Codex 懶人包 #04：第二大腦設定指南

> 版本：v0.2（Codex 版）
> 更新日期：2026-04-26
> 前置：建議先完成 #03，確認 Codex 已知道 `<VAULT_PATH>`，並已通過至少一種讀寫測試。

---

## 這個懶人包會幫你做什麼？

把一個 Obsidian vault 整理成適合 Codex 長期協作的「第二大腦」：

- 建立三層資料夾：輸入、消化、輸出
- 建立全域與 vault 內兩層 `AGENTS.md`
- 建立不依賴 Templater 的純 Markdown 模板
- 建立 `知識庫/index.md` 與 `知識庫/log.md`
- 設定 Web Clipper 或手動剪藏流程
- 建立每週知識重整 SOP
- 保留踩坑筆記，讓後面使用者避開同樣問題

> [!important]
> 本懶人包預設「補缺，不覆蓋」。
> 如果目標檔案已存在，Codex 應先讀取原檔並詢問使用者是否合併、跳過或備份後覆寫。

## 先確認：你現在是哪一種狀態？

| 狀態 | 做法 |
|------|------|
| 已完成 #03 且 MCP 可用 | 可直接照本懶人包執行 |
| 已完成 #03 但只用資料夾授權 | 可執行，但要確認 Codex 對 `<VAULT_PATH>` 有寫入權限 |
| 還不知道 vault 在哪 | 先回 #03 找到 `<VAULT_PATH>` |
| 已有成熟 vault | 務必使用「補缺，不覆蓋」模式 |
| 全新空 vault | 可照本懶人包完整建立 |

本懶人包所有路徑都用：

```text
<VAULT_PATH>
```

請把它替換成使用者實際的 Obsidian vault 路徑。

## 階段一：詢問使用者偏好

Codex 先問使用者四件事：

1. 你的職業或主要身份是什麼？
2. 你的筆記主要用途是什麼？例如教學、研究、影片創作、專案管理。
3. 希望筆記與回覆使用什麼語言？預設繁體中文。
4. 這個 vault 是否已經有重要資料？如果有，預設不覆蓋任何既有筆記。

建議記錄成：

```markdown
## 使用者偏好

- 身份：<使用者回答>
- 主要用途：<使用者回答>
- 語言：繁體中文
- 覆寫原則：不覆寫既有筆記，先讀原檔再詢問
```

## 階段二：建立三層資料夾結構

在 `<VAULT_PATH>` 建立下列資料夾。已存在就跳過。

```text
<VAULT_PATH>/
├── Clippings/           # 輸入：網路文章、影片筆記、外部資料
├── 知識庫/              # 消化：AI 整理後的結構化知識
├── 創作庫/              # 輸出：自己的教案、教材、腳本、報告
├── 每日筆記/            # 時間管理：每日紀錄、週計畫、週報
├── Templates/           # 純 Markdown 模板
└── AGENTS.md            # vault 內筆記規則
```

PowerShell 範例：

```powershell
$vault = "<VAULT_PATH>"
$folders = @("Clippings", "知識庫", "創作庫", "每日筆記", "Templates")
foreach ($folder in $folders) {
  New-Item -ItemType Directory -Force -Path (Join-Path $vault $folder) | Out-Null
}
```

資料夾語意：

| 層 | 資料夾 | 放什麼 | 原則 |
|----|--------|--------|------|
| 輸入 | `Clippings/` | 剪藏、逐字稿、外部資料 | 原始資料，少改寫 |
| 消化 | `知識庫/` | AI 整理後的主題頁 | AI 寫，人審 |
| 輸出 | `創作庫/` | 自己的作品 | 保留個人聲音 |
| 時間 | `每日筆記/` | 每日、每週、重整週報 | 記錄流程 |
| 模板 | `Templates/` | 筆記格式 | 不依賴外掛 |

## 階段三：建立兩層 AGENTS.md

### 3-1. 全域 AGENTS.md：讓 Codex 記得 vault 在哪

位置：

```text
C:\Users\<使用者>\.codex\AGENTS.md
```

macOS / Linux：

```text
~/.codex/AGENTS.md
```

建議加入：

```markdown
## Obsidian 第二大腦

主要 Obsidian Vault：

`<VAULT_PATH>`

當我說「Obsidian」、「Secondbrain」、「我的筆記本」、「第二大腦」時，預設指這個資料夾。

如果任務涉及筆記、剪藏、知識庫、創作庫、每日筆記、專案工作流程，請優先讀取：

- `<VAULT_PATH>\AGENTS.md`
- `<VAULT_PATH>\知識庫\index.md`
- `<VAULT_PATH>\知識庫\log.md`

實際讀寫仍以 Codex 工作區授權或 MCP 設定為準。
```

### 3-2. Vault 根 AGENTS.md：定義筆記規則

位置：

```text
<VAULT_PATH>\AGENTS.md
```

建議內容：

```markdown
# 我的第二大腦 — AGENTS.md

## 使用者身份

- 身份：<使用者身份>
- 主要用途：<教學 / 研究 / 創作 / 專案管理>
- 語言：繁體中文

## 資料夾規則

| 資料夾 | 用途 | 原則 |
|--------|------|------|
| `Clippings/` | 外部輸入 | 原始資料，不任意改寫 |
| `知識庫/` | 結構化知識 | AI 可協助整理，使用者審核 |
| `創作庫/` | 自己的作品 | 不改寫個人聲音，除非使用者要求 |
| `每日筆記/` | 時間紀錄 | 記錄每日、每週與重整結果 |
| `Templates/` | 模板 | 純 Markdown，避免外掛依賴 |

## 新增筆記規則

- 正式筆記要加 frontmatter：`title`、`date`、`type`、`tags`
- 優先用 Obsidian 雙向連結 `[[...]]`
- 新增知識庫頁面後，要更新 `知識庫/index.md`
- 重要操作要追加到 `知識庫/log.md`

## 安全規則

- 不覆寫既有筆記；如果檔案存在，先讀原檔再詢問
- 不刪除筆記，除非使用者明確指定
- 不批次改寫個人風格強烈的文章
- 不把 API Key、token、密碼寫進筆記

## 常用指令語意

- 「新增到筆記」：存進本 vault，並自動判斷資料夾
- 「整理成知識庫」：從 Clippings 或輸入內容建立 `知識庫/` 頁面
- 「跑一次知識重整」：執行每週知識重整 SOP
```

> [!tip]
> 全域 `AGENTS.md` 管「Codex 要去哪裡找 vault」。
> Vault 根 `AGENTS.md` 管「這個 vault 裡的筆記怎麼寫」。

如果使用者熟悉 Obsidian Templater，可以另外建立 Templater 版本模板。

> [!warning]
> 不要把含 `<% tp.xxx %>` 的示範語法放在會被 Templater 自動執行的資料夾。
> 曾遇過教學文件中的 `<% %>` 被誤展開，導致範例不可逆地變成日期或具體文字。
> 預設建議先用純 Markdown 模板。

## 階段六：設定剪藏流程

### 6-1. Web Clipper（可選）

如果使用者有安裝 Obsidian Web Clipper：

1. 點瀏覽器上的 Web Clipper 圖示

沒有 Web Clipper 也沒關係。

Codex 建議存成：


## 階段七：每週知識重整 SOP

Codex App 支援自動化。對一般使用者來說，最穩的主流程是：

1. 先用手動方式跑一次，確認流程正確。
2. 再請 Codex 建立每週自動化。
3. 若自動化失敗，仍可回到手動觸發。

> [!important]
> 不要把作業系統的 cron / Windows 工作排程器當成預設。那是舊式 CLI 路線，只有進階使用者需要。

### 7-1. 先手動跑一次（必要驗證）

每週日對 Codex 說：

```text
請跑一次我的 Obsidian 每週知識重整。
請依照 vault 的 AGENTS.md 規則執行。
不要覆寫既有筆記；若需要更新既有頁面，先摘要你要改哪裡再詢問。
```

手動成功後，確認：

- Codex 能讀取 vault
- Codex 能建立週報筆記
- `知識庫/index.md` 與 `知識庫/log.md` 的更新規則合理
- 沒有覆寫既有重要筆記

### 7-2. 建立 Codex 自動化（推薦）

手動驗證成功後，對 Codex 說：

```text
請幫我建立一個每週日早上 9:17 執行的 Codex 自動化。

自動化名稱：Obsidian 每週知識重整

工作區：<VAULT_PATH>

任務內容：
請執行 Obsidian 筆記庫的每週知識重整。
請依照 vault 根目錄 AGENTS.md 的規則執行。
不要覆寫既有筆記。
若需要更新既有頁面，先整理更新計畫。
完成後建立週報到 `每日筆記/`，並更新 `知識庫/index.md` 與 `知識庫/log.md`。
最後回報新增與修改的檔案清單。
```

Codex 會使用自己的自動化功能建立週期任務。這比叫使用者自己寫 cron 或 Windows 工作排程器更適合一般測試者。

### 7-3. AGENTS.md 主動提醒（備援）

可以在全域 `~/.codex/AGENTS.md` 加：

```markdown
## 知識重整提醒

如果今天是週日，且本週尚未做過知識重整，請在對話開始時提醒使用者：
「今天要不要跑一次 Obsidian 每週知識重整？」

只有在使用者同意後才執行。
```

這不是自動化，只是提醒規則；真正定期執行請用 Codex 自動化。

### 7-4. 重整 prompt

```text
請執行 Obsidian 筆記庫的每週知識重整。

前提：
- 使用 vault 的 AGENTS.md 規則
- 不覆寫既有筆記
- 新增或修改前先說明計畫

第一步：盤點本週變動
- 找過去 7 天新增或修改的筆記
- 分成 Clippings、創作庫、知識庫、每日筆記

第二步：整理 Clippings
- 讀本週新增 Clippings
- 分析主題與關聯
- 為每個主題建立或更新知識庫頁面
- 沒有新 Clippings 就跳過

第三步：創作庫回流
- 讀本週創作庫變動
- 若有可沉澱的概念，整理到知識庫

第四步：知識庫健康檢查
- 檢查孤兒頁面、缺少摘要、缺少來源、缺少雙向連結

第五步：更新索引與 log
- 更新 `知識庫/index.md`
- 在 `知識庫/log.md` 追加本次操作紀錄

第六步：產出週報
- 建立 `每日筆記/YYYY-WXX 知識重整.md`
- 摘要本週新增知識、可用素材、待補缺口

完成後回報新增與修改的檔案清單。
```

### 7-5. CLI 排程是什麼？為什麼不當主流程？

CLI 排程是指：

- 用 macOS / Linux 的 `cron`，或 Windows 工作排程器
- 在固定時間呼叫 `codex` CLI
- 讓作業系統定期啟動 Codex 指令

這不是 Codex Desktop 的主流程，也不是一般使用者需要理解的設定。

> [!warning]
> 只有在使用者明確說「我要用終端機 / CLI / 系統排程」時，才補充 CLI 排程。
> 對外懶人包預設請使用 Codex 自動化。

## 實測紀錄與踩坑筆記

### 踩坑 1：不要用「參考 Claude Code 版」代替完整內容

對外懶人包應該可以獨立執行。若寫「內容跟 Claude Code 版相同」，測試者會跳到另一個 repo，且可能把 `CLAUDE.md`、Skill、Claude 排程概念混進 Codex。

解法：Codex 版直接內建完整模板與完整步驟。

### 踩坑 2：Templater 不應作為預設

Templater 是 Obsidian 外掛，不是每個使用者都有裝。含 `<% tp.xxx %>` 的教學文件也可能被誤執行。

解法：預設純 Markdown；Templater 只放進階選項。

### 踩坑 3：不要把 CLI 排程當成 Codex 自動化

CLI 排程是用作業系統去呼叫 `codex` CLI，例如 cron 或 Windows 工作排程器。這跟 Codex App 內建自動化不是同一件事。

解法：公開懶人包以 Codex 自動化為推薦路線；手動觸發作為驗證與備援；CLI 排程只放進階說明。

### 踩坑 4：不要假設 Google Drive 路徑

使用者可能用 OneDrive、Obsidian Sync、iCloud、本機資料夾。

解法：全篇使用 `<VAULT_PATH>`，由 #03 先定位。

---

## 本懶人包不做的事

- 不強迫使用 Google Drive
- 不要求安裝 Templater
- 不把 CLI 排程當作預設自動化方案
- 不覆寫既有筆記
- 不沿用 Claude Code 版的 `CLAUDE.md`、Skill、Slash command 設計
